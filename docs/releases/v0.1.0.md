# Fiber Offers v0.1.0

Initial public release of Fiber Offers: reusable signed payment intents that
resolve into fresh Nervos Fiber Network invoices.

## Published packages

- [`@fiber-offers/protocol@0.1.0`](https://www.npmjs.com/package/@fiber-offers/protocol)
- [`@fiber-offers/sdk@0.1.0`](https://www.npmjs.com/package/@fiber-offers/sdk)
- [`@fiber-offers/cli@0.1.0`](https://www.npmjs.com/package/@fiber-offers/cli)

```bash
npm install @fiber-offers/protocol
npm install @fiber-offers/sdk
npm install --global @fiber-offers/cli
```

## Highlights

- Canonical signed Fiber Offer protocol with deterministic IDs and
  `fbroffer1...` Bech32m encoding.
- Fixed, flexible, reusable, single-use, CKB, token, expiry, recurrence, and
  signed revocation terms.
- Resolver integration with real merchant Fiber RPC invoice creation and
  settlement synchronization.
- Merchant CLI for initialization, node verification, offer creation,
  registration recovery, inventory, and revocation.
- SDK clients for resolvers, payer Fiber nodes, guarded payment flows,
  diagnostics, topology, recurrence, React, and React Native.
- Fiber Address discovery through `username@domain` and well-known HTTPS lookup.
- PostgreSQL, Redis, BullMQ, two resolver replicas, worker, and Nginx production
  topology.
- Merchant dashboard, payer payment pages, webhook operation, receipts, QR
  codes, and responsive in-product documentation.
- Terraform-managed AWS deployment with HTTPS at
  <https://fiber-offers.leothatguy.me>.

## Live Fiber evidence

The release was validated against real Fiber nodes, including:

- repeated resolutions producing distinct invoices and payment hashes;
- direct payment of a copied `fibt...` invoice;
- resolver synchronization from invoice creation to paid state;
- a successful public multi-hop 1 CKB payment from a payer with no direct
  merchant channel;
- a public 1,000 CKB channel providing approximately 901 CKB initial inbound
  merchant capacity after reserves.

## Current limitation

The public hosted merchant has proven inbound liquidity, but its current
liquidity counterparty is a local payer node. A durable production deployment
should move that channel or replace it with an always-on public liquidity peer.
